Si, perdido de cambios importantes puede ocurrir durante un rebase git si no se hace correctamente. el rebase es una técnica degit especial, más avanzada, y más segura que una simple merge, pero también puede ser más riesgosa. la razón principal es que el rebase reescribe manualmente la historia de commits, y debe hacerlo correctamente, sin conflictos o perdido de cambios, de lo contrario, el historial se puede ver mal. en este punto, se recomienda siempre que los cambios se empiecen a hacer después de haber seguido correctamente el repositorio original; se puede hacer backporting de las mejoras y correctores para ellos, pero no se recomienda rebasar después de haber cambiado el historial, ya que se puede perder información importante, incluso hay algún riesgo de que se pueda congelar o romper el repositorio git. es por ello que siempre se recomienda una copia de seguridad, o incluso mejorar las prácticas de versiones (gitflow, por ejemplo) o usar el workflow de feature branches, cambiando el historial solo después de que se haya emparejado en un ambiente
